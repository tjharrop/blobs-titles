Notes üìù


Not everyone is into buttons and typefaces

Different people get different benefits, but it's predictable


Accessibility is a really tangible benefit


People will want to talk about 'look and feel' but it deminishes the benefits, but find any branding people and get them on board


Presenting the needs met in a modular way helps people not to get distracted by what they consider unimportant


Talking about how all these things are connected up helps people realise how broad it is

Notes üìù

Community based design systems (created by users, too)


Sounds like an easy way to get free work, but it actually about ownership and approval


Sneaky way to offer shared ownership where people might otherwise block your progress


You can still curate content, but let others use your DS to distribute and publish their stuff


It reduces the gap between end users and the design system by a lot


It will help later when we need some cultural change


At this stage you're building the relationship, rather than gathering content

Design systems principals bring it all together


A design system principal:

  • Understands development, design, and content strategy
  • Advocates for user needs
  • Understands org needs, too
  • Makes sure the day-to-day work is done in a way that will match the bigger changes
  • Maybe the name is 'architect'?

Balancing consistency and flexibility


It doesn't matter whether its an atom, molecule, or organism

It does matter how quickly you can make it

Start with a universal baseline (styleguide)

Tailwind (et al) is your friend

Once that's live, working out component needs is easier

Create flexible foundations


Release those


Use them yourselves to create consistent components


Et voila! Both!

If it's just in figma, it's not a design system

Your design system is in code, figma helps communicate design intention

Figma doesn't use structured data, but it can with plugins

Storing the design system data in text somewhere will save a lot of maintenance headaches
